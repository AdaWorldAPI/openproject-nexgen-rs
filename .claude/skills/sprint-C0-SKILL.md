# SKILL: Sprint C0 — Codegen-Zero (First End-to-End Pipeline Run)

**Read this completely. Then read `NEXGEN_SEED.md` and `SPRINT_CODEGEN.md` in the repo root before touching any file.**

## What sprint C0 is

The first end-to-end run of the codegen pipeline from `AdaWorldAPI/openproject` (Ruby/Rails source) into this repo (`openproject-nexgen-rs`), through the `AdaWorldAPI/ruff` fork's extraction + emit pipeline. The goal is not 100% coverage — it is a **working pipeline** with a **measured coverage number** and **honest calibration reports**.

## Hard rules (non-negotiable, enforced by hooks)

1. **No `grep`/`sed`/`head`/`tail`.** The PreToolUse hook blocks these. Use `view` with `view_range`, `rg`, `cargo check`, language-native config readers.

2. **No hand-editing generated code.** Files under `crates/*/src/` after a codegen run carry the header:
   ```
   // GENERATED by ruff_python_codegen — DO NOT EDIT.
   // Modify target-spec/<file>.toml instead.
   ```
   PRs that hand-edit generated files without updating the corresponding spec entry must be rejected.

3. **Never `todo!()` in compiled paths.** Stubs use `unimplemented!("extractor-gap: see calibration/<file>.json")` with an explicit comment block pointing at the source-level pattern that failed extraction.

4. **The target-spec is the source-of-truth, not the generated code.** Agents fight in `target-spec/*.toml`. The Rust source is *output*.

5. **No deletion of `openproject-rs` references.** That repo is the calibration baseline (manual port at ~6-18% coverage). Coverage comparisons require it stays frozen.

## Definition of Done per ticket

- `cargo check` is green for every crate the ticket touched, OR every file emitted is explicitly stubbed with the `EXTRACTOR-GAP` comment block (not silently broken).
- The corresponding calibration file mentions every file the ticket emitted, with at least one line per file.
- The PR description includes:
  - Source path(s) read from openproject (Ruby/Rails)
  - Target file(s) produced
  - Number of lines emitted vs stubbed
  - Calibration warnings raised (count by category)

## Calibration acceptance (per emitted file)

Every emitted file must satisfy the five calibrate.rs lints from `ruff_python_dto_check`:

1. **`unmapped-model`** — no Rails-source model reference dropped silently
2. **`template-context-mismatch`** — view context keys match the handler
3. **`form-field-gap`** — every Rails `params[:x]` has a DTO field
4. **`output-kind-mismatch`** — Rails render-kind matches Rust return type
5. **`extractor-gap`** — unclassifiable patterns are reported AT THE SOURCE LAYER, not silently emitted

If a lint fires, the fix is in the extraction profile or target spec — never in the generated file.

## Negative examples — codegen-specific patterns to avoid

### Wrong: hand-fix a generated handler signature
```rust
// crates/op-work-packages/src/handlers/list.rs
// GENERATED by ruff_python_codegen — DO NOT EDIT.
- pub async fn list(state: State<AppState>, query: Query<ListQuery>) -> Result<Json<Vec<WorkPackage>>> {
+ pub async fn list(state: State<AppState>, query: Query<ListQuery>, user: AuthUser) -> Result<Json<Vec<WorkPackage>>> {
```

### Right: update the target spec, regenerate
```toml
# target-spec/rust-axum-sqlx.toml
[handler_signatures.list_for_tenant]
extractors = ["State<AppState>", "Query<ListQuery>", "AuthUser"]
+ # Added AuthUser extractor because Rails before_action :authenticate_user!
+ # is always present on list-for-tenant handlers in OpenProject.
```
Then regenerate. The header preserves invariants.

### Wrong: silent stub
```rust
pub async fn complex_handler() -> Result<Response> {
    todo!()  // figure this out later
}
```

### Right: explicit extractor-gap with source pointer
```rust
pub async fn complex_handler() -> Result<Response> {
    // EXTRACTOR-GAP: ruff_ruby_spo could not classify this Rails action.
    // Source: app/controllers/work_packages/bulk_controller.rb:127
    // Pattern: respond_to with both :json and :html, branching on params[:format]
    // To resolve: extend ExtractionProfile::respond_to_handler in ruff_ruby_spo
    //             to recognize format-branching pattern.
    unimplemented!("extractor-gap: see calibration/bulk_controller.json")
}
```

### Wrong: ignore the calibration report
```bash
# "It compiles, ship it"
cargo check && git commit -am "wip"
```

### Right: PR includes the calibration findings explicitly
```
PR Description:

Coverage for WorkPackage slice:
- 12 handlers emitted faithfully
- 3 handlers stubbed with extractor-gap (see calibration/work_package.json)
- 0 unmapped-model warnings
- 1 template-context-mismatch (index.html.erb @work_packages_total — handler does not provide)

Recommend extending ruff_ruby_spo to handle the @ivar-aggregation pattern (proposal in extraction-gap-proposals.md).
```

## Token budget guidance

- Phase A tickets (C0-01..03): ~30-45 min per agent.
- Phase B tickets (C0-04..07): ~60-90 min per agent. The iterative nature of emit + recalibrate makes these the heaviest.
- Phase C tickets (C0-08..10): ~30-60 min per agent. Coverage measurement reads existing artifacts; reports synthesize them.

If a ticket runs over 2× the budget, the ticket is mis-scoped — flag back to the orchestrator (Jan) and split rather than continue improvising. Pattern-drift correlates strongly with over-budget agent runs.

## What to do when uncertain

- **Extraction question** ("how should this Rails pattern be lifted to a contract?"): open a `[QUESTION]` PR against `AdaWorldAPI/ruff` with the pattern as a test fixture. Do NOT decide silently.
- **Emit question** ("how should this contract become Rust?"): write the desired output in `target-spec/`, generate, and PR. The Rust output is the proposal, the spec entry is the negotiation surface.
- **Calibration question** ("is this gap a real gap or a misunderstanding?"): err on the side of reporting the gap explicitly. False positives in calibration are far cheaper than false negatives.
